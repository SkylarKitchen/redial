# How Save Works in Redial

Redial's save pipeline is a 4-stage process: **Inline Style → Source Resolution → Tiered Search → Surgical File Write**. It doesn't add custom classes or generate new CSS — it finds the *original declaration* in your source file and surgically replaces the old value with your new one. The result is that your `.module.scss` (or `.module.css`) file looks like a human hand-edited it.

---

## Stage 1: Live Preview (`apply.ts`)

When you drag a slider, `applyInlineStyle()` sets an **inline style with `!important`** on the DOM element:

```js
(el as HTMLElement).style.setProperty(prop, value, "important");
```

This is purely a **preview mechanism** — it gives you instant visual feedback while tuning. These inline styles are:

- **Tracked in memory** — An `overrides` Map keyed by element → property → `{ initial, current, className? }`. The `className` is the edit's **provenance** (ADR-0011): the class it was applied under, absent for element-scope edits. Saving derives *where* a change lands from this record, never from the panel's toggle at save time.
- **Persisted to `localStorage`** — Debounced at 150ms, keyed by pathname, so changes survive page refreshes and HMR reloads
- **Fully undoable/redoable** — A 200-entry undo stack supports both single operations and batched changes (e.g., paste)

At this point, **no source files have been touched**. Everything is ephemeral and reversible.

### Key APIs

| Function | Purpose |
|---|---|
| `applyInlineStyle(el, prop, value, className?)` | Set an inline override and track it (with its provenance) |
| `diff(el)` | Get list of `{ prop, from, to, state?, breakpoint?, className? }` changes for an element |
| `undo()` / `redo()` | Step through the change history |
| `reset(el)` | Remove all overrides for an element |
| `isDirty(el, prop)` | Check if a property has been modified (used for amber indicators) |

## Stage 2: Source Resolution (`core/save.ts` → `commitUtils.ts` → `sourcemap.ts`)

When you hit **Save** (Footer button, Cmd+S, palette) or **Save All** (ChangesDrawer), the surface collects its diffs and hands them to `save(entries)` in `core/save.ts` — THE save pipeline. It owns enrichment (`enrichChangesForCommit(element, changes)` — targeting flows from each entry's recorded provenance), the file-vs-clipboard partition, per-mode transport (a mixed batch POSTs CSS and Tailwind changes separately, since the route dispatches the whole request on `body.mode`), the no-endpoint/unreachable clipboard fallbacks, and post-save breakpoint reconciliation. Each change routes by what it carries: class provenance → the shared rule (or `.class:state` / `@media` block); element provenance (stateless, breakpointless) → an `elementScope` descriptor the server writes into the element's JSX `style` attribute. Source files are resolved with two strategies, in priority order:

### Strategy 1: CSS Modules Class Name Parsing

Looks at the element's `className` string for patterns generated by bundlers:

- **webpack**: `Button_btn__a8f2k` → derives `Button.module.scss`
- **Turbopack**: `page-module__IiFEKa__btnPrimary` → derives `page.module.css` or `.module.scss` (detected from stylesheet URL)
- **Vite**: `_btn_1a2b3_5` → derives `*.module.css` or `.module.scss` (detected from stylesheet URL)

This tells us *which file* to edit, but not the exact line number.

### Strategy 2: React Fiber `_debugSource`

Falls back to walking the React fiber tree (available in dev mode) to find `_debugSource.fileName` and `_debugSource.lineNumber` injected by SWC/Babel. This gives us the JSX file (e.g., `Button.tsx:12`) — useful as a hint for inline styles or when CSS Modules aren't in play.

### What Gets Sent to the Server

Each change is enriched into:

```ts
{
  prop: "font-size",
  from: "14px",
  to: "16px",
  sourceFile: "Button.module.scss",  // from CSS class parsing
  sourceLine: 8,                      // from source map or fiber
  className: "btn",                   // the CSS class name
  componentName: "Button",            // the React component name
}
```

## Stage 3: Server-Side Commit (`src/server/commit.ts`)

The enriched changes are POST'd to an API endpoint (default: `/api/tuner/commit`). The server performs two main operations:

### File Resolution

1. Try the direct/relative path first
2. If not found, recursively search the project by filename (excluding `node_modules`, `.next`, `dist`, `.git`, `build`, `out`, `.turbo`)
3. If multiple matches, prefer the one containing the component name
4. As a last resort, try `ComponentName.module.scss` and `ComponentName.module.css` variants

Path traversal is blocked — `..` segments and paths escaping the project root are rejected.

### Tiered Property Search

Once the file is found, a 4-tier search locates the exact line:

| Tier | Strategy | How It Works |
|---|---|---|
| 1 | **Window** | Search ±5 lines around the source map line number |
| 1.5 | **Root block** | For custom properties (`--*`), search inside `:root {}` or `[data-theme] {}` blocks |
| 2 | **Class block** | Find the `.className { }` block and search within it (brace-depth aware) |
| 3 | **Full file** | Search the entire file for `prop` + `value` together |
| 4 | **Fuzzy** | Find any line matching `prop:` — handles SCSS variables, `calc()`, etc. |

### Surgical Replacement

Only the **value** is replaced. Indentation, comments, semicolons, and surrounding code are preserved:

```scss
// Before
.btn {
  font-size: 14px;
}

// After
.btn {
  font-size: 16px;
}
```

For fuzzy matches (where the literal `from` value isn't on the line — e.g., SCSS variables), the entire value portion after `prop:` is replaced.

Changes to the same file are batched into a single write.

---

## Stage 4: HMR Reconciliation (`apply.ts`)

After `writeFile()` modifies the source, the dev server's file watcher triggers HMR. The updated stylesheet reloads in the browser, and `clearRedundantOverrides()` runs:

1. **Remove** all inline overrides from the DOM (batched write)
2. **Read** the new computed styles (single reflow)
3. **Compare** — if the real stylesheet value now matches the override, the override is discarded permanently
4. **Restore** any overrides that haven't been absorbed yet

This is carefully batched to avoid N forced reflows (one read phase, not one per property).

Once HMR catches up, the inline `style="..."` is gone. The element runs purely on the real stylesheet.

---

## Summary

| Stage | Where | What Happens |
|---|---|---|
| **Preview** | Browser (apply.ts) | Inline style with `!important` for instant feedback |
| **Resolve** | Browser (sourcemap.ts) | Figure out which source file and line to edit |
| **Commit** | Server (commit.ts) | Find the property in the file and surgically replace the value |
| **Reconcile** | Browser (apply.ts) | HMR reloads the stylesheet; inline overrides are cleaned up |

**The end result**: your `.module.scss` file reads like a human edited it. No custom classes, no leftover inline styles, no generated CSS.

---

## Configuration

The commit endpoint is configurable:

```tsx
<Tuner commitEndpoint="/api/tuner/commit" />
```

Or programmatically:

```ts
import { configure } from "redial/config";
configure({ commitEndpoint: "/custom/api/path" });
```

---

## Session Persistence

Overrides are saved to `localStorage` under `__tuner_session:<pathname>` so they survive:

- Page refreshes
- HMR reloads
- Browser tab restores

The session is capped at 100KB and automatically cleaned up when overrides are reset.
